# DBLP 边关系：各步骤边数说明

说明从原始数据到最终图，**每一步的边数**以及**各边类型分别有多少条**。  
（没有任何一步会“新增”边，预处理只是给边打类型，子图会减少边。）

---

## 步骤 0：原始数据（link.dat）

- **路径**: `data/raw/DBLP_four_area/link.dat`
- **格式**: 每行一条边，`src dst`（空格分隔），**没有边类型**
- **边数**: **36,645** 条

边类型要在下一步根据节点类型推断。

---

## 步骤 1：预处理（preprocess_dblp_rule_based.py）

- **输入**: `node.dat` + `link.dat`
- **输出**: `data/DBLP_four_area_processed/nodes.jsonl`、`edges.jsonl`
- **逻辑**: 对 `link.dat` 的每一行 `(u, v)`，查 `node.dat` 得到 `u`、`v` 的节点类型，用 `infer_edge_type(src_type, dst_type)` 得到边类型，写出到 `edges.jsonl`。
- **边数**: 与原始一致，**36,645** 条（一条原始边对应一条带类型的边）。

**按边类型划分（本步骤“打出”的类型）：**

| 边类型 (type)       | 条数   | 说明           |
|---------------------|--------|----------------|
| **author_paper**    | 9,215  | 作者–论文      |
| **paper_conference**| 2,988  | 论文–会议      |
| **paper_term**      | 24,442 | 论文–词        |
| **合计**            | 36,645 |                |

即：**本步骤没有增加或减少边，只是给每条边标上了上述三种类型之一。**

---

## 步骤 2：子图（build_dblp_subgraph.py，可选）

- **输入**: `data/DBLP_four_area_processed/nodes.jsonl`、`edges.jsonl`
- **输出**: 例如 `data/DBLP_subgraph_2hop/nodes.jsonl`、`edges.jsonl`
- **逻辑**: 从某个种子节点做 k 跳 BFS，只保留在子图内的节点，再只保留**两端都在子图内**的边。
- **边数**: 只减不增。例如 2-hop 子图：**3,257** 条边。

**2-hop 子图内按边类型：**

| 边类型 (type)       | 条数 | 说明     |
|---------------------|------|----------|
| **author_paper**    | 19   | 作者–论文 |
| **paper_conference**| 712  | 论文–会议 |
| **paper_term**      | 2,526| 论文–词   |
| **合计**            | 3,257|          |

---

## 小结

| 步骤 | 输出/位置           | 总边数 | author_paper | paper_conference | paper_term |
|------|--------------------|--------|--------------|------------------|------------|
| 0    | link.dat           | 36,645 | （无类型）   |                  |            |
| 1    | edges.jsonl（全图）| 36,645 | 9,215        | 2,988            | 24,442     |
| 2    | 2-hop 子图         | 3,257  | 19           | 712              | 2,526      |

**“哪个步骤增加了多少边”**：  
- 步骤 1：边数不变，只是给 36,645 条边分别打上类型（9,215 + 2,988 + 24,442）。  
- 步骤 2：边数减少到 3,257，没有增加边。

若你之后改了预处理或子图脚本，只要再跑一遍并重新统计上述三个位置的边数和类型即可更新本文档。
