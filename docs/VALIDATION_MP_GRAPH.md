# 验证阶段「计算图 = 噪声边 + 查询边」的含义

## 验证在做什么

验证时对**同一张图**做评估：

1. 对真实图 **加噪**，得到当前时刻的「噪声图」（边可能被删/被改类型，或新增一些噪声边）。
2. **采样一批「查询边」**：从所有可能边里按比例选出一部分，作为「我们要模型预测的边」。
3. 在这批查询边上算损失（预测 vs 真实），得到 val NLL 等指标。

也就是说：**不是**整图每条边都预测，而是每轮只**抽查**一部分边（由 `edge_fraction` 控制），多轮覆盖整图。

---

## 为什么 MP 图要包含「噪声边 + 查询边」

模型做的是**边预测**：对给定的边 (u, v)，预测它的类型（或存在性）。  
边预测依赖**图上的消息传递（MP）**：要预测 (u,v)，需要知道 u、v 的邻居和当前图上有哪些边。

因此，做一次 forward 时，模型看到的图必须同时包含：

| 部分 | 含义 | 在计算图里的作用 |
|------|------|------------------|
| **噪声边** (clean_edge_index / edge_index_t) | 加噪后的图上有哪些边、边类型是什么 | 提供「当前图结构」和边上特征，MP 时沿这些边聚合信息 |
| **查询边** (query_edge_index) | 本轮要预测的那批边 (u,v) | 这些 (u,v) 也要作为「边」放进图里，模型才能在每条边上输出预测；输入时先给一个占位类型（如 no-edge），标签用真实的 |

所以：

- **计算图 = 噪声边 ∪ 查询边**（合并去重后就是 `comp_edge_index`）。
- 边上特征：噪声边用加噪后的边类型；查询边在输入里用「占位」（如 one-hot 的 no-edge），这样模型知道「这些边是待预测的」。
- 前向结束后，用 `query_mask` 从输出里只取出「查询边」上的预测，和真实标签算损失。

**简单说**：  
噪声边 = 模型看到的当前图；查询边 = 模型要在上面做预测的边。  
两者都要放进同一张 MP 图里，模型才能既利用图结构，又只在查询边上算验证损失。

## 问题出在哪儿（逻辑链）

1. **get_computational_graph** 的输入是：
   - `clean_edge_index` / `clean_edge_attr`：加噪后的边（即 sparse_noisy_data 的 edge_index_t / edge_attr_t）；
   - `triu_query_edge_index`：本轮采样的查询边（上三角或异质图下的有向边）。

2. 函数内部：
   - 把噪声边和查询边 **拼在一起** → `comp_edge_index`；
   - 用 **coalesce** 合并重复的 (u,v)（同一条边可能既在噪声图里又在查询集里），并得到合并后的边特征和 `query_mask`（哪些位置来自「查询边」）；
   - 最后用 **one_hot(..., num_classes=de)** 得到 `comp_edge_attr`。

3. **崩溃原因**（你遇到的情况）可能包括：
   - **空边**：某轮里 `comp_edge_index` 为空（例如查询边为空且噪声图也无边），coalesce 推断 `num_nodes=0`，在 PyG/CUDA 里触发 assert；
   - **one_hot 越界**：合并后某条边的类型索引 ≥ `de` 或 < 0，在 GPU 上触发 device-side assert。

因此修复做了两件事：  
(1) 当 `comp_edge_index` 为空时直接返回，不调用 coalesce；  
(2) 对合并后的类型索引做 `clamp(0, de-1)` 再送进 one_hot，避免越界。

---

## 小结

- **含义**：验证时「计算图 = 噪声边 + 查询边」是为了让模型**在同一张图上**既看到当前结构（噪声边），又在要评估的边（查询边）上做预测。
- **问题逻辑**：在把这两部分合并、去重、转成 one-hot 边特征时，遇到空图或非法类型索引，在 coalesce/one_hot 里触发 CUDA assert；通过空图提前返回和索引 clamp 可以避免这类崩溃。
