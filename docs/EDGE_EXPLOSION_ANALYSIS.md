# 边数爆炸与「只学会加边、没学会去边」分析

## 1. 初始边数是否与真实边数一致？

**结论：一致。**

- 采样时 z_T（初始噪声图）由 `sample_sparse_discrete_feature_noise_heterogeneous_fixed_nodes` 生成。
- 边数来自 `dataset_info.edge_family_avg_edge_counts`，即**训练集上每关系族的平均边数**（见 `diffusion_utils.py` 约 969–1054 行；`dblp_single_dataset.py` 约 248–251、350 行）。
- DBLP 单图：训练集只有 1 张图，故该平均即该图边数（36645）。
- 终端日志中 `t_norm=1.0000` 时「当前显式边数=36645」，与真实边数一致，**初始化没有问题**。

---

## 2. 边数为何爆炸？Merge 逻辑里有没有「去边」？

### 2.1 更新公式（`diffusion_model_sparse.py` 约 5212–5247 行）

每内层循环（每块 query）的更新为：

- **新显式边** = （当前显式边 **减去** 本块 query 里的边）**并上** （本块 query 里 **预测为存在** 的边）

即：

```text
new_explicit = (current_explicit \ query_set_this_iter) ∪ {e ∈ query_set_this_iter : pred(e)=存在}
```

因此：

- **会去边**：当某条边在「当前图」里且落在**本块 query** 中，且模型预测为「不存在」时，该边会被从 current 中剔除，且不会在 pred_exist 里被加回。
- **会加边**：本块 query 里预测为「存在」的边会被加入。

所以从机制上讲，**模型是有机会学会「去边」的**——只要在 query 里把「该删的边」预测成 0 即可。

### 2.2 为何实际表现为「只加边、不去边」？

核心是**每条边上「加/删」的规模不对称**：

- 每块 query 大小 ≈ 全部可能边 × edge_fraction（或 1/k）。例如 493140×2% ≈ 9863 条。
- 「当前图」的边数从 36k 起步，随步数还会涨；但**本块 query 与当前边的交集**只是这 9863 条里的一小部分（随机 chunk 与 36k 的交集期望很小）。
- 因此：
  - **可删除的上界** ≈ 本块 query 与当前边的交集，通常就几百条量级；
  - **可增加的上界** ≈ 本块 query 里预测为 1 的数量，可达几千（若模型 20% 预测存在，则 ~2000）。
- 于是每块往往是：**去掉几百、加上几千 → 净增大量边**。多步、多块累积后就会出现你看到的 36k → 1.44M 的爆炸。

所以：不是「代码没有去边逻辑」，而是**在「每块 query 随机、且只对当前块做预测」的设定下，可删的边数远小于可加的边数**，模型若再偏向预测「存在」，就会表现为「只会加、不会减」。

## 3. 训练里是否只学了「加边」、没学好「去边」？

### 3.1 目标（ideal / true）是否对称

- 使用 `structure_only_global` 时，目标为 ideal：在「只改 query、固定 query 边数」的约束下，让全图结构最接近真实图 A；ideal 由对 query 上 soft 做梯度下降得到（`diffusion_model_sparse.py` 理想块）。
- 真实图 A 里，绝大多数可能边是 0、少数是 1。ideal 会尽量贴近 A，因此**在 query 上也是「大部分 0、少部分 1」**，不会只鼓励加边。
- 所以从目标定义上，**加边和去边是对称的**：该是 0 的位置学 0，该是 1 的位置学 1。

### 3.2 训练 vs 采样的结构性差异（train/sample mismatch）

- **训练**：每步只有一个 **固定的** 加噪图 + **一块** query；loss 只在这块 query 上算；**不会**把本步预测 merge 回图再给下一步用。
- **采样**：同一扩散步内有 **50 个内层循环**，每块 query 预测完后会 **立刻 merge**（若开启 `sampling_update_graph_per_inner_loop`），下一块用的是「上一块更新后的图」。

因此：

- 训练时模型从未见过「当前图已经比 z_T 多出很多边」的输入；
- 采样时从第 2 块开始，输入图就比训练时任意一步的图都更密，且越来越密。
- 在这种**更密、且从未在训练中出现过的输入分布**上，模型容易表现异常——例如更倾向预测「存在」，从而加剧边数增长。  
这可以部分解释：**不是目标没教「去边」，而是采样时的输入分布和训练不一致，导致行为偏向「加边」**。

### 3.3 度分布（subtype_degree）为何难对齐？

- 度分布损失 `_subtype_degree_hist_loss`（`train_metrics.py` 约 654–742 行）是在**全图**上算的：
  - `true_deg`：真实全图或 full graph 上的真实度；
  - `pred_deg`：**当前步**的「全图」度——即把 **noisy 图在 query 位置用模型预测替换** 后的图的度（`_compute_structure_stats` 等用的是 full 图 + query 处 pred）。
- 因此梯度只会通过「全图度分布」反传到 **本块 query 上的预测**；且度是全局聚合，单块 query 只占可能边的一小部分，**对整体度分布的梯度信号较弱**。
- 若再叠加：
  - 只做结构、不做边 CE（structure_only）时，没有「逐边 0/1」的强监督；
  - 或 ratio loss 下其他结构项（relation_matrix、metapath2/3）权重更大；
  则**度分布项容易被压过**，训练时就会「控制不好度分布的对齐」，采样时边数更容易失控。

## 4. 小结与建议

| 问题 | 结论 |
|------|------|
| 初始边数是否和真实一致？ | 一致；z_T 用训练集平均边数（单图即 36645）。 |
| 有没有「去边」机制？ | 有；当前图里、且在本块 query 中、且预测为 0 的边会被删。 |
| 为何看起来像「只加边」？ | 每块可删的边（当前边 ∩ query）远少于可加的边（query 里 pred=1）；模型若再偏 1，就净增。 |
| 训练是否只学了加边？ | 目标对称（0/1 都学）；但训练时从未见过「已 merge 多块、图很密」的输入，采样分布偏移导致偏向加边。 |
| 度分布为何难对齐？ | 度损失在全图上、梯度只经 query；信号弱，且易被其他结构损失盖过。 |

**建议（可择一或组合）：**

1. **训练阶段**
   - **平衡加边/去边**：开启 `model.train_query_balance_add_remove: true`，使每块 query 中约一半为「该加」(真实有、噪声无)、一半为「该删」(噪声有、真实无)，训练情景在边数上接近中性，与采样 merge 时的加删对称一致；不同 t 下模型已学不同混乱程度，主要补齐边数量上的匹配。
   - 适当**加大 `subtype_degree_loss_weight`**（如 0.2→0.3），或单独做度权重的 ablations。
   - 在 structure_only 时也保留一定**边存在性监督**（或弱 CE），让「该 0 的位置」有更直接的梯度。
   - 若可行：在训练中**模拟「多块 merge」**（例如每步多块 query、每块后更新 comp 再算下一块），使训练分布更接近采样。

2. **采样阶段**
   - **减小跳步**（如 `sampling_skip=5` 或 1），减轻单步跨度带来的预测偏差和边数暴涨。
   - 对「预测为存在」的边做**后处理截断**：例如每步或最终只保留 top-k 条（按概率或分数），或按目标边数/度约束截断，防止显式边数无界增长。

3. **诊断**
   - 在采样时打日志：每块内 **pred=0 且当前图有该边** 的数量（实际删掉的边）、**pred=1** 的数量（新加的边），确认「去边」确实发生且比例是否过小。
   - 在训练中记录 **subtype_degree_L1** 与 relation_matrix / metapath 的比值，确认度损失是否过弱。

## 5. 是否应该硬性限制边数？（与 ideal 边数不变对齐）

### 5.1 现状

- **训练**：`structure_only_global` 下 ideal 在 query 上**边数不变**（只优化「哪 k 条为 1」，k = 噪声图在 query 上的存在边数），训练目标是「边数中性」。
- **采样**：没有硬约束。每块 query 的更新是「先删掉当前图中落在本块 query 的边，再按模型预测把本块里预测为存在的边加回去」。预测为存在的条数可以远多于被删的条数 → **边数会增长**（你看到的 36645 → 43029 再继续涨）。

所以：**训练目标已经是「边数不变」，但采样没有施加同样的约束，二者不一致。**

### 5.2 结论：建议在采样阶段加「边数不变」硬约束

**应该限制**，且更合适的是在**采样阶段**做硬约束，和「ideal 边数不变」对齐：

1. 训练里 ideal 已经假设「只调哪 k 条为 1、不改变边数」，模型是在这个设定下学的；采样时若不限制，模型输出的「存在」条数会自由膨胀，和训练假设不符。
2. 在采样里做「每块边数不变」或「每步总边数不变」，不改变已训练好的模型参数，只改变**如何从预测中选出边**，实现简单、风险小。
3. 若只在训练里加辅助 loss（如 pred 在 query 上的存在边数 ≈ noisy），可能和结构损失冲突，且仍无法保证采样时每块/每步边数不涨，不如采样时直接约束。

### 5.3 实现思路（采样阶段）

在 `sample_p_zs_given_zt` 里，对**每一块 query** 做完模型预测和后验后，在得到 `sampled_edge_attr`（或决定「哪些 query 位置为存在」）时加一层约束：

- **本块边数不变**：设 `k = 当前图中落在本块 query 的边数`（即本块会被删掉的边数）。在本块 query 上，只允许 **k 条**为「存在」：
  - 用后验或模型给出的 P(存在)（或 P(非 0)）对 query 位置排序，取 **top-k** 为存在，其余置为 0（no-edge）。
  - 这样本块「删 k 条、加 k 条」→ 该块对总边数贡献为 0，多块累积后总边数保持与上一步一致；若从 z_T 起就保持目标边数，则整条轨迹边数不变。

可选配置：如 `general.sampling_fixed_edge_count_per_block: true` 时启用上述逻辑；为兼容现有行为可默认 false。

**备选**：若不做「每块」约束，可以做**每步结束后**的总边数截断（例如按得分保留 top 36645 条），也能抑制爆炸，但不如每块边数不变与 ideal 设定一致。
